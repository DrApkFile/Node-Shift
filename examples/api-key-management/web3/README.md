# API Key Management — Web3 Implementation

> **Pattern:** API Key Management | **Chain:** Solana Devnet | **Framework:** Anchor

---

## How This Works in Web2

In Web2, API keys are long-lived secrets generated by a server, stored in a database, and shared with the client. The server verifies the key on every request using HMAC or a database lookup. If the key leaks, it must be regenerated.

```javascript
// Typical Web2 API key check
const key = req.headers["x-api-key"];
if (!apiKeys.has(key)) return res.status(401).json({ error: "invalid key" });
```

**Key Web2 components:**
- A `api_keys` database table storing secret key strings.
- Server middleware performing O(1) set-lookup or HMAC verification.
- A key rotation and revocation process managed by the platform.

---

## How This Works on Solana

On Solana, the concept of a "secret key" is replaced by **cryptographic identity**. Instead of sharing a secret, a user's wallet address (their public key) is registered on-chain in a `RegistryEntry` PDA. Authorization is proven by the user's *cryptographic signature* on the transaction itself — no secret needs to be stored or transmitted.

```
Admin ──register_key──▶ Program ──creates──▶ RegistryEntry PDA (authorized_pubkey = user's key)

User ──authorized_call──▶ Program ──derives──▶ RegistryEntry PDA (seeds = [b"api_key", user.key()])
                              verify:   entry.is_active == true
                              verify:   user's signature (from transaction signer)
                                  ✅ → Execute | ❌ → Revert
```

The user never shares a secret. The only "proof" of authorization is their ability to sign with their private key — which never leaves their wallet.

---

## Key On-Chain Mechanics

- **`register_key`**: Admin creates a `RegistryEntry` PDA seeded by `[b"api_key", target_user.key()]`. Sets `authorized_pubkey`, `is_active = true`, and a human-readable `label`.
- **`revoke_key`**: Admin sets `is_active = false` on the registry PDA. All future calls from this wallet will be rejected.
- **`authorized_call`**: Any wallet can call this instruction with their `user` key and the correspodning PDA. The Anchor account constraint automatically:
  1. Derives the correct PDA from the `user` signer's key.
  2. Verifies `entry.is_active == true` — if false, reverts with `RevokedKey`.

## Architecture Diagram

```
Admin ──register──▶ Program ──creates──▶ RegistryEntry PDA
                                          { authorized_pubkey: user.key(), is_active: true }

Admin ──revoke──▶ Program ──updates──▶ RegistryEntry PDA { is_active: false }

User ──authorized_call──▶ Program:
  1. Derive PDA: [b"api_key", user.key()]
  2. Load RegistryEntry
  3. require!(entry.is_active) → Execute or Revert
```

## Tradeoffs & Constraints

| Aspect | Web2 | Solana |
|--------|------|--------|
| **Secret Handling** | Shared secret key (leakable, phishable) | Public key + signature (private key never shared) |
| **Revocation** | Private server event; user has no proof of when or why | On-chain transaction; verifiable timestamp and reason |
| **Scale** | Millions of keys trivially in a database | Each key = one PDA (~0.001 SOL rent). Cost-prohibitive for mass-market free APIs |
| **Key Types** | Any capability (read, write, admin) | Encoded via separate PDAs or role fields per authorization level |

**Key Constraints:**
- **Admin Centralization**: The `register_key` and `revoke_key` instructions are called by an `admin`. For a decentralized app, this admin should be a multi-sig or DAO-controlled wallet.
- **No Key Expiry**: The current implementation has no `expires_at` field. Production use should add an expiry timestamp checked in `authorized_call`.
- **Closing Revoked Keys**: A revoked PDA with `is_active = false` still occupies rent. A `close_key` instruction should be added to reclaim rent from revoked entries.
